# 03. Rapid Logging（高速ログ）— BuJo の言語

## このドキュメントで分かること

- Rapid Logging が「BuJo の言語」と呼ばれる意味
- 4 つの構成要素（Topic / Page Number / Short Sentences / Bullets）
- 公式の書き方規則と、実例
- なぜ「短文」と「素早さ」が同時に要求されるのか

**想定読者**: BuJo の全体像は掴んだが、実際の「書き方」の文法を知らない実装者。データ入力 UI を設計する人。

---

## 3.1 Rapid Logging とは何か

### 公式定義

> "Rapid logging is the language in which the Bullet Journal method is written. Rapid logging involves quickly jotting down information in a concise and structured manner with the use of Bullets."[^rl-faq]

[^rl-faq]: <https://bulletjournal.com/blogs/faq/what-is-rapid-logging-understand-rapid-logging-bullets-and-signifiers>

ここでのキーフレーズは三つ。

1. **The language**: 単なる「書き方」ではなく、BuJo を成立させる文法
2. **Quickly**: 素早さ。Carroll は「数秒で書ける」ことを重視している
3. **Concise and structured**: 短く、かつ構造化されている

つまり Rapid Logging は、「**頭の中に浮かんだものを、3 秒で構造化して紙に落とす**」ための文法装置である。

### なぜ「Rapid」なのか

ADHD 起源の文脈を思い出すと、これは必然である。

- 頭の中に思考が次々と湧く（雨が降ってくる）
- 文章で書こうとすると、書いている間に次の思考が押し寄せて忘れる
- だから「短文 + 記号」で**1 行 3 秒**で書ける形式が必要だった

長文日記が続かない人でも、「`・` 牛乳を買う」と書くだけならできる。これがメソッドの実用上の核。

---

## 3.2 Rapid Logging の 4 構成要素

Carroll の書籍と公式サイトでは、Rapid Logging は次の 4 つで構成される、と教える[^how-to][^rl-faq]。

[^how-to]: <https://bulletjournal.com/pages/how-to-bullet-journal>（リダイレクト経由でアクセス確認、`/pages/learn` は 404 になることがある）

```
┌──────────────────────────────────────────────────────┐
│ 1. Topic（トピック）                                 │
│    ページ上端に書く「このページのタイトル」          │
│    例: "May 2026"  "Daily Log - 2026-05-24"          │
└──────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────┐
│ 2. Page Number（ページ番号）                         │
│    すべてのページに番号を振る                        │
│    Index と Threading で参照される                   │
└──────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────┐
│ 3. Short Sentences（短文）                           │
│    1 つのエントリ = 1 行（できれば）                 │
│    意味が分かる最小限の語数で                        │
└──────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────┐
│ 4. Bullets（記号）                                   │
│    各行の頭に種類を示す記号を置く                    │
│    Task: ・  Event: ○  Note: –                       │
└──────────────────────────────────────────────────────┘
```

順に詳述する。

---

## 3.3 構成要素 1: Topic（トピック）

### 役割

各ページの一番上に、**そのページが何のためのページか**を書く。インデックスと連動して、後で見返したときに「これは何だっけ？」を防ぐ。

### 公式の指示

> "At the top of the page, record the date as your topic."[^how-to]

これは Daily Log の場合の例だが、原則として**どのページにも Topic がある**。

- Daily Log: 日付（例 `5.24.26`、または `2026-05-24`、または `5月24日 月`）
- Monthly Log: 月名（例 `May 2026` または `5月 2026`）
- Future Log: そのスパン名（例 `Future Log: Jun–Nov 2026`）
- Custom Collection: 任意のテーマ名（例 `Books to Read`、`Travel Plan Italy`、`Project X`）

### Topic の機能

Topic はインデックスへの登録対象である。例えば「Books to Read」というトピックでページを使い始めたら、それを Index に「Books to Read … p.42」と登録する。

---

## 3.4 構成要素 2: Page Number（ページ番号）

### 役割

各ページに番号を振る。これによって、

- Index から「Books to Read … p.42」のように参照できる
- Threading（後述）で「p.42 → p.78 → p.103」のように離れたページを繋げる
- 過去のページを「あの日のこと」「あの月の振り返り」として参照できる

### 公式ノート（Leuchtturm1917 Edition 2）の対応

Leuchtturm1917 と Carroll がコラボした公式ノート Edition 2 では、**ページ番号が中央配置**になっている。これは「threading で見返すときに、見開きで番号を読みやすくするため」と公式は説明している[^leuchtturm]。

[^leuchtturm]: <https://www.leuchtturm1917.us/bullet-journal-edition-2.html>

### デジタル化の論点

ページ番号は紙の物理的な順序を前提とした概念。デジタルでは「タイムスタンプ」「UUID」「日付」のいずれかで代替できるが、

- **時系列を保つ**: 紙のページ順 = 時系列なので、これはデジタルで自然に再現可能
- **見返しの単位**: ページ単位だった概念が、「日」「月」「コレクション」のどの単位に置き換わるか
- **Threading の代替**: ハイパーリンクで自然に置き換わる

詳細は `07-threading-and-index.md` および `10-digital-implementation-considerations.md`。

## 3.5 構成要素 3: Short Sentences（短文）

### 公式の指示

> "Short-form sentences paired with bullets that serve as visual anchors."[^rl-faq]

各エントリは、**1 行で完結する短文**であるべきとされる。

### 例

良い例:
```
・ 牛乳を買う
・ プロジェクト A の議事録を共有
○ チームミーティング 14:00
– Andreas のフィードバック: 文書化が弱い
```

悪い例（長すぎる、構造化されていない）:
```
・ 今日はプロジェクト A の議事録を作って、それを Slack で共有して、それから牛乳も買って、あと Andreas からフィードバックがあったので対応する
```

### なぜ短文か

書籍によれば、長文を書こうとすると次の問題が起きる。

1. **書く時間が伸びる**: 「Rapid」じゃなくなる
2. **続かない**: 毎日 30 分の日記は維持しづらい
3. **検索性が落ちる**: 後で見返したとき、何が書いてあるか把握しづらい
4. **思考のスピードに追いつかない**: 1 文書いている間に次のアイデアを忘れる

### 反省したいときは Custom Collection

「もっと書きたい」「考えを練りたい」場合は、Daily Log には書かず、専用の Custom Collection（journaling page）を別ページに作り、そちらで自由に書く。これが BuJo の柔軟性。

## 3.6 構成要素 4: Bullets（記号）

これが Rapid Logging の華。詳細は `04-signifiers-and-bullets.md` に譲るが、ここでは Rapid Logging の文脈での位置づけだけ。

### 3 つの基本 Bullet

| 記号 | 名前 | 意味 |
|---|---|---|
| `・` | Task（タスク） | 自分が行動を起こすべきもの |
| `○` | Event（イベント） | 日時に紐づく出来事 |
| `–` | Note（ノート） | 情報、考え、観察 |

> "Tasks are represented by a simple dot '•'. Events are represented by an 'O' Bullet. Notes are represented with a dash '–'."[^rl-faq]

### なぜ 3 種類なのか

Carroll は、**4 種類以上は覚えきれない**という制約から、3 種類に絞ったと書籍で説明している。これは ADHD 設計の制約でもある:

- 2 種類: 区別が粗すぎる
- 4 種類: 書くたびに「これはどれだっけ？」と迷う
- 3 種類: 直感的に判別可能（やる / 起きる / メモする）

これは UI 設計でいう「**Hick の法則**」（選択肢が多いほど決定時間が伸びる）を、Carroll が経験則で発見していた格好。

### 記号は「視覚的アンカー」

公式の表現:

> "Bullets that serve as visual anchors."[^rl-faq]

`・` `○` `–` は、ページをめくったときに視覚的にひっかかる「アンカー」として機能する。3 種類のシルエットが違うので、

- ページを 1 秒見ただけで「今日はタスクが多い日」「ノートばかりの日」が分かる
- 一括スキャンで「未完了タスク」（`・` の状態）を見つけられる
- パラパラめくっても情報密度を把握できる

## 3.7 Rapid Logging の典型的なフロー

実際の運用イメージを順序立てる。

### 朝（AM Reflection）

```
1. ノートを開く
2. 今日のページ（Daily Log）を作る
3. 上端に Topic を書く（例: "5/24 月"）
4. ページ番号を振る
5. （必要なら）Monthly Log を見て、今日やるべきことを書き写す
6. （必要なら）前日の未完了タスクをマイグレーション
7. 心の中の雑念を Rapid Logging で外部化
```

### 日中（随時）

```
新しい思考・タスク・イベントが浮かぶ
   ↓
ノートを開く
   ↓
Bullet を選ぶ（・ / ○ / –）
   ↓
短文で書く
   ↓
（必要なら）Signifier を頭に追加（*, ! 等）
   ↓
閉じる
```

ここでのポイントは、「**判断が 3 つしかない**」こと（タスクか、イベントか、ノートか）。これが意思決定を高速化する。

### 夕方〜夜（PM Reflection）

```
1. 今日のページを開く
2. 完了したタスクの・を×に変える
3. やらなかったタスクを見て、判断:
   - 続ける → 翌日の Daily Log に > で migrate（PM ではしないことも）
   - やめる → 打ち消し線
   - 後回し → Future Log に < で schedule
4. （任意）今日の気づきを Note (–) で追記
5. 明日の Daily Log の準備
```

PM Reflection の詳細は `06-migration-and-reflection.md`。

## 3.8 Rapid Logging のリアルな例（Daily Log）

公式と複数の二次情報から再構成した、典型的な Daily Log の例:

```
─────────────────────────────────────
            5.24.26 (月)
─────────────────────────────────────
・ プロジェクト A 議事録を Slack へ
× 牛乳を買う
・ チームミーティングの議題を準備
*・ Andreas からのフィードバックに返信
○ 14:00 チームミーティング
○ 19:30 ヨガクラス
– Andreas のコメント: 「ドキュメントが薄い」
– 朝の散歩で気づいた: 自分の時間を最優先する
!– React Server Components で書き直す案
>・ 確定申告の書類を準備
                                p.42
─────────────────────────────────────
```

読み方:

- `・` 未完了タスク
- `×` 完了したタスク（牛乳買えた）
- `*・` 優先タスク（Signifier `*` 付き）
- `○` その日のイベント（時間つき）
- `–` ノート（メモや観察）
- `!–` インスピレーション（Signifier `!` 付きの Note）
- `>・` 翌日（または翌期間）に migrate された未完了タスク

## 3.9 Rapid Logging の「失敗パターン」

書籍と公式ブログから、ありがちな失敗を列挙。これは UX 上の警戒対象でもある。

### 失敗 1: 長文化

ノートを開くたびに自由作文をしてしまうと、Rapid Logging ではなくただの日記になる。これは続かない。

→ **対策**: 「1 行ルール」を意識する。長く書きたいなら Custom Collection を作る。

### 失敗 2: Bullet を忘れる

行頭に `・` `○` `–` を書き忘れると、視覚的アンカーが機能しなくなる。

→ **対策**: ペンを取ったら必ず Bullet から書き始める癖をつける。

### 失敗 3: 完璧主義（特に装飾系）

「綺麗に書こう」「コーディネートしよう」と思った瞬間、Rapid じゃなくなる。装飾系 BuJo の罠。

→ **対策**: 公式は「Rapid」の語に「速い」「ラフ」「未完成でいい」というニュアンスを込めている。

### 失敗 4: 全部 Task にしてしまう

何でも `・` で書いてしまい、Event や Note が消える。これだと「予定」が見えなくなる。

→ **対策**: 「これは行動か、出来事か、情報か」を毎回 1 秒考える癖。

### 失敗 5: マイグレーションを溜め込む

Rapid Logging は書くだけで、見返さない、移行しない、と、ノイズが溜まって機能不全になる（→ `06`）。

## 3.10 Rapid Logging とリスト系アプリの違い

ここで、デジタル化を考える上で重要な比較を置く。

| 項目 | Rapid Logging | Todoist 等のタスクアプリ |
|---|---|---|
| データ型 | Task / Event / Note の混在 | ほぼ Task のみ |
| 入力時間 | 数秒（手書き） | 数秒（タイプ） |
| 視覚アンカー | `・` `○` `–` の形状 | チェックボックス、フラグ等 |
| 順序 | 時系列固定（書いた順） | 並び替え自由 |
| 編集 | 訂正できるが、書いたことは残る | 自由に編集・削除 |
| メタデータ | Topic + Page Number のみ | 期限、ラベル、優先度等が豊富 |
| 哲学 | 思考の外部化 | タスクの遂行管理 |

ここで気づくのは、**Rapid Logging は「タスクアプリ」よりも「3 種類のエントリが時系列に混ざるストリーム」に近い**ことだ。これは X（旧 Twitter）のタイムラインや、Slack の自分宛チャンネルに近い構造である。

デジタル化のヒント:

- **時系列ストリーム** UI が自然
- **3 種のシルエットの違い**を視覚的に保つ（チェックボックス、・、ダッシュ等）
- **編集 UI の摩擦**を保つ（自由に並び替えできすぎると哲学が壊れる）

## 3.11 Rapid Logging と「短期記憶 → 長期記憶」の橋渡し

書籍によれば、Rapid Logging は脳科学的にも理にかなっている。

- 頭の中の思考は**短期記憶**（容量限界 7±2 項目）に保持される
- 容量を超えると、新しい思考が古い思考を押し出す（=忘れる）
- Rapid Logging で外部化すると、短期記憶の容量を解放できる
- 紙に書かれた情報は、いつでも参照可能な**外部メモリ**として機能する
- 翌日や翌月に見返すことで、**長期記憶**にも定着しやすい

この機構は、デジタル化でも維持可能。ただし、

- **検索可能性に頼りすぎると見返さなくなる**（Evernote の罠）
- **能動的な見返し（リフレクション）**を促す UI が必要

## 3.12 アプリ実装への含意

| 観点 | 含意 |
|---|---|
| 入力速度 | クイック入力は最重要。1 タップで Bullet 種別、1 入力で完了する UI |
| 文字数制限 | 緩い制限（200 字程度？）で短文を促す。長文は別画面に誘導 |
| 3 種類の Bullet 選択 | Task / Event / Note の 3 択を、迷わず選べる UI（Hick の法則） |
| 視覚アンカー | 3 種の Bullet を視覚的にはっきり差別化（色 / アイコン / 形状） |
| 時系列順序 | 入力順を固定。並び替え機能は最小限 |
| 編集 | 編集は可だが、削除は「打ち消し線」のような可逆表現にする選択肢 |
| Topic | エントリ単位ではなくページ／日付単位の概念。データモデルに分離 |
| Page Number | ID は持つが UI 表示は最小限。Threading の代替を別途設計 |

## 3.13 まとめ

1. Rapid Logging は **BuJo の言語**。短文 + 記号で「思考を秒で外部化する」文法
2. 4 要素: **Topic / Page Number / Short Sentences / Bullets**
3. 3 種類の Bullet（`・` `○` `–`）は、ADHD 設計から来る「**最小の判別単位**」
4. 「速さ」と「短さ」は哲学に直結する（雑念を整流する装置）
5. デジタル化では「**入力の速さ**」と「**3 種の視覚的区別**」を死守する必要がある

次は `04-signifiers-and-bullets.md` で、記号体系の完全な仕様と状態遷移を扱う。

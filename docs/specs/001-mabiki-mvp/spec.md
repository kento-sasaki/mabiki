# Feature Spec: Mabiki MVP — コア体験「捨てる」と「移行する」

**Feature ID**: 001
**Branch**: `001-mabiki-mvp`
**Status**: Draft（レビュー前）

## 1. 概要（What）

Mabiki MVP は、デジタル Bullet Journal における **タスクの「整理（Mabiki）」プロセス** のコア体験を提供する。一般的なタスク管理アプリが追求する自動化・効率化をあえて排除し、手書きが持つ「面倒くささ（摩擦）」をインタラクションで再現することで、ユーザーに **「これは本当に今日やるべきか？」** の再考を促す。

MVP で検証する2つのコア・インタラクション:

1. **捨てる（Mabiki）** — 不要と判断したタスクを、ハサミで切り落とすような演出を経て即座に消す。
2. **移行する（Migration）** — 翌日に持ち越すタスクを、ワンタップやコピペではなく **目視しながら手で再入力** して引き継ぐ。

## 2. 背景・狙い（Why）

手書き BuJo では、未完了タスクを翌月・翌日に **書き写す（Migration）** 行為そのものが「本当に必要か」を問い直すフィルタになる。デジタルで自動繰り越しにすると、この問い直しが消え、タスクが「ただのノイズ」として蓄積する。

Mabiki は、この **意図的な不便さ（デジタル摩擦）** をプロダクトのコア価値に据える。MVP の目的は、この摩擦が「心地よい再考の儀式」になるのか「ただのストレス」なのかを、開発者自身のドッグフーディングで体感・検証することにある。

---

## 3. スコープ

### In Scope（MVP に含む）

- 当日のデイリーログ（タスク一覧）の参照
- タスクの完了操作（摩擦なし）
- 「捨てる（Mabiki）」操作 — 演出付き・即時消滅
- 「移行（Migration）」操作 — コピペ禁止・手動再入力
- 振り返り（イブニングリチュアル）モード — 前日以前の未完了タスクの仕分け

### Out of Scope（MVP に含めない）

- 自動繰り越し（**意図的に実装しない**）
- 一括移動・ドラッグ&ドロップでの並べ替え（**意図的に実装しない**）
- Future Log / Monthly Log / Index / Threading / Collections
- 記号体系（Signifiers）の網羅的サポート
- マルチデバイス同期・共有

## 4. ユーザーストーリーと受入基準

### US-1: 日中のタスク完了（摩擦なし）

> ユーザーとして、本日のデイリーログを開き、完了したタスクを摩擦なくチェックしたい。

**受入基準**
- [ ] 本日のデイリーログ画面に、本日分の未完了・完了タスクが表示される
- [ ] タスクをタップ／チェックすると即座に「完了」状態になり、取り消し線などの視覚フィードバックが返る
- [ ] 完了操作には摩擦（確認・再入力など）を入れない

### US-2: 捨てる（Mabiki）

> ユーザーとして、不要になったタスクを潔く・一発で消したい。

**受入基準**
- [ ] タスクの「削除（×）」を実行すると、ハサミで切り落とすような演出が再生される
- [ ] 演出後、タスクは画面から消滅する
- [ ] 削除は復元できない（潔さの演出。データ層は `deletedAt` による論理削除だが、UI から復元手段は提供しない — §7 で確定）

### US-3: 移行する（Migration）

> ユーザーとして、翌日に持ち越すタスクを、手で打ち直すことで「本当にやりたいか」を再確認したい。

**受入基準**
- [ ] 移行対象タスクに対して「移行（>）」を選ぶと、今日の入力エリアにフォーカスが当たる
- [ ] 元タスクの内容は **参照用としてうっすら表示** される（入力欄は空のまま）
- [ ] 入力欄でのペースト（Cmd/Ctrl+V）は **ブロック** され、手動タイピングを強制する
- [ ] 入力を確定すると、本日のタスクとして新規登録される
- [ ] 移行元タスクは「移行済み（Migrated）」状態になり、編集不可になる

### US-4: イブニングリチュアル（振り返りと仕分け）

> ユーザーとして、1日の終わりに前日以前の未完了タスクを1件ずつ仕分けし、思考を整理したい。

**受入基準**
- [ ] 振り返りモードに入ると、前日以前の未完了タスクが古い順に表示される
- [ ] 各タスクを「移行（US-3）」または「捨てる（US-2）」で1件ずつ仕分けできる
- [ ] 日中のタスク消化モードと、振り返りモードは UI 上・体験上分離されている
- [ ] 仕分けが完了すると本日のデイリーログへ遷移する
- [ ] **例外**: 前日以前の未完了タスクが0件の場合、仕分けステップはスキップして本日のデイリーログを表示する

## 5. デジタル摩擦の設計原則

| 操作 | 一般的な Todo アプリ | Mabiki の摩擦設計 | 狙い |
|---|---|---|---|
| 日付の切り替わり | 未完了を自動繰り越し／期限切れ表示 | **自動繰り越ししない。** 未完了は元の日に留まる | 自動化された蓄積（ノイズ化）を防ぎ、再考を促す |
| 移行 | 「明日へ」ワンタップ／D&D | **コピペ禁止で全く同じ内容を再タイピング** | 「打ち直してまでやりたいか」を脳に再確認させる |
| 削除 | ゴミ箱に入り後で復元可 | **演出と共に一発で消滅** | 物質的な「ちぎって捨てる」潔さを再現 |
| 振り返りの場 | タスク一覧内でインライン操作 | **振り返りモードという時間・空間（UI）の分離** | 消化モードからリフレクションモードへの精神的切り替え |

### 摩擦設計の研究根拠

各摩擦は推測ではなく `docs/research/` の哲学的根拠に紐づく（[Constitution 原則 3](../../guides/constitution.md)）。

- **自動繰り越ししない**
  - [research/02-philosophy.md §2.5「効率化の罠」](../../research/02-philosophy.md) — 自動繰り越しは BuJo の「signal を noise から分離する装置」を破壊する
  - [research/10-digital-implementation-considerations.md §10.4「意図的な摩擦をデジタルでどう再現するか」](../../research/10-digital-implementation-considerations.md) / §10.14「アンチパターン集」
- **コピペ禁止・再タイピング**
  - [research/06-migration-and-reflection.md §6.1「Migration の核は『書き写しの摩擦』」](../../research/06-migration-and-reflection.md) — Migration は "surface what's worth the effort" を目的とする儀式
  - [research/02-philosophy.md §2.5「書き写しの摩擦が生む選別効果」](../../research/02-philosophy.md) / [§2.8「手書きの認知科学」](../../research/02-philosophy.md)（Mueller & Oppenheimer 2014）— 「自分の言葉で再構築する」プロセス自体が再考を生む
- **演出付きの一発消滅（復元なし）**
  - [research/02-philosophy.md §2.8「アナログによる『意図的な摩擦』— 哲学の物質的根拠」](../../research/02-philosophy.md) — 物質的な「ちぎって捨てる」潔さ。「ゴミ箱に入れて後で復元」は判断の先送りを許してしまう
- **振り返りモードの時間・空間的分離**
  - [research/02-philosophy.md §2.3 Mindfulness / §2.4 ストア哲学](../../research/02-philosophy.md) — 朝の準備モードと夜の振り返りモードを別画面として実装する設計示唆
  - [research/06-migration-and-reflection.md AM/PM Reflection](../../research/06-migration-and-reflection.md) — 日次リフレクションは「注意の方向づけ／振り返り」を担う装置

## 6. 非機能要件

- スマホファースト（スマートフォン幅でのタップしやすさ・見え方を最優先）
- 摩擦は「移行」「削除」の意思決定点に限定し、完了・参照は軽快に保つ
- 演出（切り落とし）はストレスにならない速度・質感であること

## 7. Clarifications（解消済み）

SDD 原則に従い、ドラフト間で食い違う／未確定の点は推測で埋めずに明示し、Clarify フェーズで解消した結果を以下に記録する。決定は [docs/guides/constitution.md](../../guides/constitution.md) に照らして判断した。

- [x] **削除の方式** → **復元不可の一方向論理削除（`deletedAt` で隠す）** に確定。
  - 根拠: UI 上は復元不可（[Constitution 原則 1](../../guides/constitution.md) の「潔さ」を保つ）。データ層に痕跡を残すのは、誤削除のトラブルシュートおよび将来の利用パターン分析の余地を残すため。UI が復元を提供しない限り、哲学的要請は満たされる。
- [x] **移行の一致判定** → **完全一致を強制しない**（納得した内容で確定可）。
  - 根拠: コピペ禁止＋手動再入力という摩擦が機能していれば「打ち直してまでやりたいか」の問い直しは成立する。完全一致の強制は誤字修正・要約・言い換えを阻害し、[research §2.8（手書きの認知科学）](../../research/02-philosophy.md) の「自分の言葉で再構築する」効果に逆行する。
- [x] **イブニングリチュアルの強制度** → **任意起動**。
  - 根拠: MVP の主眼は「摩擦の触感」の体感検証（§2）。まず任意で運用し、ドッグフーディングで「強制したほうが定着しそう」と判断したら段階的に強制度を上げる（[research §2.6 Kaizen](../../research/02-philosophy.md)）。
- [x] **ユーザー/認証** → **シングルユーザー**（ドッグフーディング前提）。
  - 根拠: §3 Out of Scope の「マルチデバイス同期・共有」と整合。`userId` 列は将来のために保持するが、認証機構は組まない。
- [x] **バレット種別** → **Task のみ**。
  - 根拠: §3 Out of Scope の「記号体系（Signifiers）の網羅的サポート」と整合。コア摩擦（捨てる・移行する）の検証に Note/Event は不要。
- [x] **移行先の指定** → **常に「今日」に限定**。
  - 根拠: 任意日付への先送りを許すと、意思決定の先送り（「とりあえず明日に、そして明々日へ」）を許してしまい、[Constitution 原則 1](../../guides/constitution.md) と矛盾する。BuJo の Daily Log 哲学とも一致。
